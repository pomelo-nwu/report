# 技术布道报告生成器

深度分析给定项目，生成**面向开发者的技术布道报告**。

## 原则

1. **像布道者，不像文档工程师**——每一章都要有节奏、有感染力、有洞察，不是信息的搬运
2. **深入浅出**——一句话讲清概念，代码/图解再撑住细节；不堆砌术语，不跳过关键
3. **论据驱动**——每个判断必须有代码片段、设计决策或数据支撑；禁止空洞赞美（"这个项目很棒"→ 必须说"X 设计解决了 Y 问题，方式是 Z"）
4. **对比出真知**——与同类方案对比，突出取舍而非优劣；读者需要坐标系，不是排行榜
5. **读者是中级开发者**——默认读者有工程经验但未必熟悉该领域；该解释的术语要解释，该省的入门常识要省

---

## 写作节奏

每章自然落入这个节奏，不必刻意套模板：

- **开篇抓住人**：反直觉的事实、极端案例、或"直觉上你会觉得X，但实际上Y"——让读者停下来
- **用故事而非罗列**：具体场景展开，让读者身临其境；叙事 > 要点列表
- **提炼可迁移洞察**：不是"它做了X"，而是"X背后的设计哲学是Y，这对你的项目意味着Z"
- **给行动出口**：读者读完能做什么？思考题、动手实验、可复用的代码片段

**语气**：第一人称视角（"让我们来看…"、"你会发现…"）；对真正出色的设计可适度惊叹，但必须附证据；不用"颠覆性"、"革命性"，用"更务实"、"更精准"等事实描述。

## 执行步骤

### 1. 获取源材料

**GitHub 仓库：**
```bash
mkdir -p docs/<项目名>
git clone <项目地址>.git docs/<项目名>/.repo
```

**网页链接：**
- web_fetch 抓取内容，提取核心信息
- 如引用了 GitHub 仓库，一并 clone 到 `docs/<项目名>/.repo/`
- 所有原始材料存入 `docs/<项目名>/`

### 2. 深度研读

**广度扫描：**
- README / docs / 核心页面 → 项目定位、核心主张、目标用户
- 配置文件（package.json / requirements.txt / Cargo.toml 等）→ 技术栈与依赖
- 目录结构 → 模块划分、架构骨架

**深度挖掘：**
- 核心入口文件（main.py / index.ts / src/ 等）→ 主流程与关键路径
- 测试文件 → 设计意图、边界场景
- 抽取 2-3 个代表性模块深入分析

**布道素材：**
- git log / changelog → 关键里程碑
- 贡献者分布、issue 讨论、设计文档 → 项目背后的"人"与"运动"
- 同类项目/论文 → 对比坐标系

### 3. 自由结构，自然叙事

**不预设固定章节模板。**根据项目的独特性，自己决定：

- 章节数量（建议 4-5 篇，不超过 6 篇）
- 每篇的主题和标题（标题要反映该章的独特视角，不用模板词）
- 由浅入深的排列顺序
- 篇幅分配（核心章节可以长，速览章节必须短）

唯一约束：**第一篇必须是速览**——一句话定位 + 3 分钟读懂核心价值，篇幅精炼不拖沓。最后一篇必须是展望——局限性、替代方案、未来方向、开放问题。

**文件命名**：`docs/<项目名>/XX-具体标题.md`（XX 为序号，标题反映独特视角）

**如果项目涉及 AI Agent / Code Agent / AI Coding 等正在重塑开发者工作方式的领域**，在相关章节中自然嵌入行业影响分析——该项目出现前后能力边界的变化、对开发者工作方式的具体影响、人机协作建议——不要单独开一篇，让它在叙事中有机出现。

## 产出规范

- 技术术语首次出现时简要解释
- 代码片段标注**为什么重要** + 来源文件路径，而非仅仅展示
- 适当使用表格、ASCII 图示辅助理解
- 对比分析用表格，不用纯文字
- 每章末尾给出 **Key Takeaways**——不超过 5 条，每条一句话
